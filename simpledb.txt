Simple JSONL Database
=====================

A schema-agnostic in-memory database that communicates via stdin/stdout.
Records are stored as JSON Lines (one JSON object per line).
All field values are strings. The schema is strict: records must contain
exactly the expected fields, no more, no less.


Startup Sequence
----------------

The first command must be "expect", which defines the schema.
The second command should be "load", which loads an existing file
or starts empty if the file doesn't exist yet.

    expect ["name","city","age","path"]
    load /path/to/data.jsonl


Commands
--------

expect <field_list>
    Must be the first command. Defines the expected fields as a JSON
    array of strings. All subsequent add/load operations validate
    against these fields (both missing and extra fields are rejected).

    Example:  expect ["name","city","age"]
    Response: {"ok":true}

load <filepath>
    Loads records from a JSONL file. If the file does not exist, starts
    with an empty record set (the file will be created on save). Each
    record is validated against the expected fields. If any record fails
    validation, the load is rejected and the previous state is preserved.

    Example:  load /home/user/data.jsonl
    Response: {"ok":true,"count":250000}

save
    Writes all records back to the file specified in the load command.

    Example:  save
    Response: {"ok":true}

add <json_record>
    Adds a new record. Must contain exactly the expected fields.
    If a sort is active, the record is inserted in sorted position;
    otherwise it is appended.

    Example:  add {"name":"Alice","city":"London","age":"30"}
    Response: {"ok":true}

count
    Returns the number of records currently in memory.

    Example:  count
    Response: {"ok":true,"count":250000}

clear
    Deletes all records in memory. The file path and schema are
    retained, so add and save still work afterwards.

    Example:  clear
    Response: {"ok":true}

select <json_filter>
    Returns all records matching the filter. Each filter field is
    matched as a case-insensitive substring (like SQL's %term%).
    Multiple fields are combined with AND logic. An empty filter
    matches all records.

    The response is multi-line: a status line with the count, followed
    by one line per matching record.

    Example:  select {"name":"smith","city":"london"}
    Response: {"ok":true,"count":2}
              {"name":"John Smith","city":"London","age":"42"}
              {"name":"Jane Smithson","city":"East London","age":"35"}

    Example:  select {}
    Response: (returns all records)

    Pair matching (see below) can also be used in select filters.

delete <json_filter>
    Deletes all records matching the filter. Matching works the same
    as select, including pair matching. Returns the number deleted.

    Example:  delete {"name":"John Smith"}
    Response: {"ok":true,"count":1}

sort <field>
    Sorts all records by the given field (case-insensitive). From
    this point on, records remain sorted: add inserts in sorted
    position via binary search rather than appending.

    A new sort replaces the previous one. Load resets the sort.

    Example:  sort "relpath"
    Response: {"ok":true}

deleteone <json_filter>
    Deletes the first matching record and returns. Matching works
    the same as select and delete, including pair matching.

    Internally, the search starts at the index of the last deleteone
    hit, wrapping around if needed. This means that if records are
    sorted and deletions arrive in sorted order, each deleteone is
    nearly instant since the search picks up right where it left off.

    Example:  deleteone {"relpath":"games/abc.sgf"}
    Response: {"ok":true,"count":1}

    If no match is found:
    Response: {"ok":true,"count":0}

quit
    Exits the program immediately. Does not auto-save.


Sorting and Batch Deletion
--------------------------

For best performance when deleting many records, sort first, then
send deleteone commands in the same sorted order:

    sort "relpath"
    deleteone {"relpath":"aaa/file1.sgf"}
    deleteone {"relpath":"aaa/file2.sgf"}
    deleteone {"relpath":"bbb/file3.sgf"}
    ...
    save

This makes the entire batch roughly O(n) rather than O(n*m).


Pair Matching
-------------

The special key "__pair__" can be included in a select or delete
filter to match two fields where the values are interchangeable.
The value is an array of two arrays: field names and search values.

    "__pair__": [["fieldA", "fieldB"], ["value1", "value2"]]

Two values provided:
    Matches if (fieldA=value1 AND fieldB=value2) OR
               (fieldA=value2 AND fieldB=value1).

    Example:  select {"__pair__": [["PB","PW"], ["Alice","Bob"]]}
    Finds games where Alice played Black and Bob played White,
    or Alice played White and Bob played Black.

One value provided:
    Matches if fieldA=value OR fieldB=value.

    Example:  select {"__pair__": [["PB","PW"], ["Alice"]]}
    Finds all games where Alice played either color.

Zero values:
    Omit "__pair__" entirely; use a normal filter.

Pair matching is combined with AND logic against any other fields
in the same filter:

    select {"__pair__": [["PB","PW"], ["Alice","Bob"]], "EV": "title"}
    Finds games between Alice and Bob in events containing "title".


Error Responses
---------------

Any command can return an error instead of a success response:

    {"error":"missing field \"age\""}
    {"error":"unexpected field \"bogus\""}


Notes
-----

- All matching is case-insensitive substring (contains).
- Delete can remove multiple records if the filter matches broadly.
- Save overwrites the file entirely each time.
- There is no auto-save; call save explicitly after mutations.
- A failed load does not alter the current in-memory state.
- Commands other than expect, load, and quit require a successful
  load to have been completed first.
